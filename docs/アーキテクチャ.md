# NotchHub アーキテクチャ設計規約

Swift + SwiftUI / AppKit による macOS 常駐アプリのアーキテクチャ設計規約。
**MVVM + Service + Repository** のレイヤード構成を採用し、機能（Feature）単位でモジュールを分割する。

---

## ディレクトリ構成

```text
Sources/NotchHub/
├── App/                    # エントリポイント・常駐管理
│   ├── NotchHubApp.swift   # @main / NSApplicationDelegate
│   ├── MenuBar/            # メニューバー / Launch at Login
│   └── Composition/        # DI 組み立て（Composition Root）
├── Core/                   # 全機能共通の基盤
│   ├── Models/             # ドメインモデル（外部依存なし）
│   ├── Extensions/         # 標準型の拡張
│   ├── Utilities/          # 純粋ユーティリティ
│   └── DI/                 # 依存解決の型・コンテナ
├── Features/               # 機能単位モジュール（View + ViewModel）
│   ├── Notch/              # ノッチ UI・展開状態・最小状態
│   ├── Shelf/              # 一時保管
│   ├── AirDrop/            # AirDrop 起動・履歴
│   ├── Share/              # Share Sheet
│   ├── AIMonitor/          # Claude / Codex / Antigravity
│   ├── Calendar/           # 予定表示
│   └── Media/              # メディア操作
├── Services/               # ビジネスロジック（protocol + 実装）
├── Repositories/           # 永続化（protocol + 実装 + Stub）
├── Platform/               # OS 統合（protocol + 実装 + Stub）
└── Resources/              # アセット・Info.plist 等
```

---

## レイヤー構造

| 層 | ディレクトリ | 責務 |
|---|---|---|
| View | `Features/*/Views/` | SwiftUI ビュー。表示と入力のみ。ロジックを持たない |
| ViewModel | `Features/*/ViewModels/` | 画面状態（`@Observable` / `ObservableObject`）。Service を呼び出し View へ状態を供給 |
| Service | `Services/` | ビジネスロジック・複数 Repository/Platform の協調。protocol 経由で依存を受け取る |
| Repository | `Repositories/` | 永続化（SQLite・BookmarkData・設定）。protocol を実装 |
| Platform | `Platform/` | OS API ラッパー（Sharing・EventKit・Accessibility・Socket・AppleScript）。protocol を実装 |
| Model | `Core/Models/` | ドメインモデル。外部依存なし。最内層 |

---

## 依存方向（厳守）

```text
View → ViewModel → Service → (Repository protocol / Platform protocol)
                                        ↑
                            Repository 実装 / Platform 実装
Model（Core）は何にも依存しない
```

- **Model（Core/Models）は最内層。** 他層・フレームワーク（SwiftUI/AppKit/SQLite）への依存禁止
- **View は ViewModel にのみ依存。** Service / Repository / Platform を直接参照しない
- **ViewModel は Service（protocol）にのみ依存。** Repository / Platform を直接参照しない
- **Service は Repository / Platform の protocol にのみ依存。** 具体実装を直接 import しない
- **Repository / Platform の実装は Core/Models と外部 API に依存。** 定義した protocol を実装する
- **DI 組み立ては `App/Composition/` でのみ行う。** 唯一、全層を参照してよい場所

## View / ViewModel 規約

SwiftUI と AppKit（ノッチウィンドウ・グローバルドラッグ検出）の接続点。

| 種別 | 責務 |
|---|---|
| View | レイアウト・表示・ユーザー入力の受け取り。状態は ViewModel から受け取る |
| ViewModel | 画面状態の保持と更新。Service を呼んで結果を `@Published` / `@Observable` プロパティへ反映 |

- **View にビジネスロジックを書かない。** 分岐・整形は ViewModel または Service に置く
- **ViewModel は Service を protocol 型で受け取る（コンストラクタ DI）。** テスト時に Stub へ差し替え可能にする
- **ノッチウィンドウ・グローバルイベント監視は AppKit（NSWindow / NSPanel / NSEvent monitor）で実装し、Platform 層に隠蔽する**

---

## Service 実装規約

**全 Service は protocol を定義し、`final class` + コンストラクタ DI で実装する。** グローバル関数・シングルトンによるロジック実装は禁止（DI で注入する `AppEnvironment` 等は例外）。

- 依存（Repository / Platform / 他 Service）は `init` で `private let` として受け取る
- 公開メソッドで機能を提供し、ヘルパーは `private` に閉じる
- 単一モデルで完結するロジックはモデルのメソッドに置く。Service には「複数モデル/複数依存の協調」のみを置く

例外: 値型（`struct` / `enum`）のドメインモデル・ユーティリティ・`Result` ベースの型定義は自由に定義・公開してよい。

---

## ドメインモデル設計

**Rich Domain Model を徹底する（Anemic Model 禁止）。**

- 値の生成・検証・変換はモデル自身のイニシャライザ / メソッドで行う
- 不変条件はイニシャライザで強制し、不正な状態のインスタンス生成を許さない（failable init / throwing init を活用）
- ドメインモデルは値型（`struct` / `enum`）を基本とし、SwiftUI/AppKit/SQLite への依存を持たない

---

## Repository / Platform パターン

外部依存（SQLite・ファイルシステム・OS API）は必ず protocol で抽象化し、Service は protocol 経由でのみアクセスする。

| 種類 | protocol 配置 | 実装配置 |
|---|---|---|
| Repository | `Repositories/` | `Repositories/`（本番実装 + `Stub`） |
| Platform | `Platform/` | `Platform/`（本番実装 + `Stub`） |

- **本番実装と Stub 実装を必ずペアで用意する。** Stub はテスト・プレビュー・権限未付与時のフォールバックに使う
- **どの実装を使うかは `App/Composition/` で組み立てる。** ViewModel / Service は実装の存在を知らない
- 永続化の詳細（テーブル設計・BookmarkData）は [`リポジトリ層設計規約.md`](リポジトリ層設計規約.md) を参照

## モジュール境界の機械的検証

- レイヤー違反（View → Repository 直接参照、Service → 実装 import 等）を避けるため、依存方向はレビューで必ず確認する
- 可能なら Feature 単位を Swift Package のローカルターゲットに分割し、`import` をコンパイル時に制約する
- 詳細な品質チェックは [`品質チェック・テスト規約.md`](品質チェック・テスト規約.md) を参照
