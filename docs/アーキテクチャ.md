# {{PROJECT_NAME / 例：〇〇プロジェクト}} アーキテクチャ設計規約

{{FRAMEWORK_ENV / 例：pnpm workspace + Next.js App Router}} における {{ARCHITECTURE_STYLE / 例：DDD バックエンド}} 設計規約。

## ディレクトリ構成

```text
{{TARGET_DIR / 例：src/backend/}}
├── domain/
│   ├── models/          # ドメインモデル
│   ├── services/        # ドメインサービス
│   ├── gateways/        # Gateway interface
│   └── repositories/    # Repository interface
├── application/
│   └── usecases/        # ユースケースオーケストレーション
├── infrastructure/
│   ├── adapters/        # Gateway 実装（AI, 外部 API 等）
│   ├── repositories/    # Repository 実装
│   └── db/              # DB 接続プール等
└── presentation/
    ├── composition/     # Composition Root（DI 組み立て）
    ├── loaders/         # データ取得（読み取り専用）
    └── actions/         # 副作用（CUD, Server Actions）
```

## {{ARCHITECTURE_STYLE / 例：DDD}} 4 層構造

| 層 | ディレクトリ | 責務 |
|---|---|---|
| presentation | `presentation/composition/` `loaders/` `actions/` | {{PRESENTATION_ROLE / 例：Next.js Server Components / Server Actions との接続}} |
| application | `application/usecases/` | ユースケースオーケストレーション。{{APP_SPECIFIC_LOGIC / 例：プロンプト生成・レスポンスパース含む}} |
| domain | `domain/models/` `services/` `gateways/` `repositories/` | ビジネスルール。外部依存なし |
| infrastructure | `infrastructure/adapters/` `repositories/` `db/` | Gateway / Repository 実装。外部サービス接続 |

---

## 依存方向（厳守）

`presentation → application → domain ← infrastructure`

- **domain は最内層。** 他層への依存禁止
- **application は domain にのみ依存。** infrastructure への直接依存禁止（Gateway interface 経由）
- **infrastructure は domain/gateways, domain/repositories に依存。** interface を implements する
- **presentation/loaders, actions は application + presentation/composition にのみ依存。** domain・infrastructure の直接参照禁止
- **presentation/composition は全層を参照可。** UseCase と infrastructure 実装を組み立てる唯一の場所
- **フロントエンド** (`{{FRONTEND_DIR / 例：app/, frontend/}}`) **は backend/presentation/ にのみ依存**

---

## presentation 層: composition / loaders / actions

{{FRAMEWORK_NAME / 例：Next.js}} と {{ARCHITECTURE_STYLE / 例：DDD}} の唯一の接続点。**{{API_RULE / 例：API Route は原則使用しない。}}**

| 種別 | 責務 | 呼び出し元 |
|---|---|---|
| composition | UseCase + infrastructure 実装の組み立て（Composition Root） | loader, action |
| loader | データ取得（読み取り専用） | Server Component (`{{PAGE_COMPONENT_PATTERN / 例：page.tsx}}`) |
| action | 副作用（CUD） | Server Actions (`{{SERVER_ACTION_PATTERN / 例：'use server'}}`) |

- **{{PAGE_COMPONENT_PATTERN}} は loader を呼んで props を渡すだけ。** ロジック禁止
- **loader / action は composition から UseCase を取得して呼ぶ。** infrastructure の直接 import 禁止
- **composition は全層を参照してよい唯一の場所。** UseCase に infrastructure 実装を注入し、エクスポートする

---

## ドメインモデル設計

**{{DOMAIN_MODEL_RULE / 例：Rich Domain Model を徹底する。Anemic Domain Model 禁止。}}**

- 値の生成・検証・変換はモデル自身のファクトリメソッド / インスタンスメソッドで行う
- ドメイン不変条件はファクトリメソッド / コンストラクタで強制する。不正な状態のオブジェクト生成を許さない
- domain service に置くのは「複数モデルの協調」のみ。単一モデルで完結するロジックをサービスに書かない

---

## サービス実装規約

**全サービス（domain service, UseCase）はクラスベース + コンストラクタ DI。関数エクスポートによるサービス実装は禁止。**

- コンストラクタの `private readonly` パラメータで Gateway 等の依存を受け取る
- ビジネスロジックは `execute()` 等の public メソッドで公開する
- ヘルパーは private メソッドに閉じる。クラス外への export 禁止

例外: 型定義 (`interface`, `type`) およびユーティリティ型 (`{{UTILITY_TYPES / 例：Result<T,E>}}` 等) は関数 / 定数エクスポート可。
