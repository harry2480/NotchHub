# NotchHub Repository レイヤー設計方針

永続化（SQLite・BookmarkData・設定）へのアクセスは `Repositories/` に集約する。Service（ビジネスロジック）からは必ず Repository protocol を介してデータを操作し、SQL や OS のファイル API を直接触らない。

## 基本パターン

protocol を `Repositories/` に定義し、本番実装と Stub を同居させる。

```swift
// Repositories/ShelfRepository.swift

protocol ShelfRepository {
    func fetchAll() throws -> [ShelfItem]
    func search(query: String) throws -> [ShelfItem]
    func insert(_ item: ShelfItem) throws
    func delete(id: ShelfItem.ID) throws
    func setPinned(id: ShelfItem.ID, pinned: Bool) throws
}

// 本番実装（SQLite）
final class SQLiteShelfRepository: ShelfRepository {
    private let db: DatabaseConnection
    init(db: DatabaseConnection) { self.db = db }

    func fetchAll() throws -> [ShelfItem] {
        // SELECT ... ORDER BY pinned DESC, created_at DESC
    }
    // ...
}

// テスト・プレビュー用 Stub
final class StubShelfRepository: ShelfRepository {
    private var items: [ShelfItem]
    init(items: [ShelfItem] = []) { self.items = items }
    // インメモリ実装
}
```

## 実装ルール

- **Repository は `App/Composition/` から注入された DB 抽象（`DatabaseConnection`）を利用する。** DB 実装を Repository 内や Service / ViewModel で生成しない
- **Repository は「クエリ実行 + 第一段のエラーハンドリング」のみを担当する。** 複雑なビジネスロジック（上限超過時の削除判定・寿命判定）は Service に置く
  - 例外: DB で完結する集計・ソートは Repository 側で実装してよい
- **戻り値はドメインモデル（`Core/Models`）に変換して返す。** SQLite の行・型を上位層へ漏らさない
- **本番実装と Stub をペアで用意する。** `App/Composition/`（`AppComposition`）を唯一の Composition Root とし、どちらを注入するかをそこで切り替える

## データストアと責務

| Repository | ストア | 内容 |
|---|---|---|
| `ShelfRepository` | `shelf.db` | Shelf アイテム（参照保持メタ・テキスト本文） |
| `SettingsRepository` | `settings.db` | 設定（寿命・上限・初期タブ・各種表示 ON/OFF） |
| `AirDropHistoryRepository` | `airdrop_history.db` | AirDrop 履歴（名前・種類・日時・元パス・結果。送信先は保存しない） |

保存先ルートは `~/Library/Application Support/NotchHub/`（[`インフラストラクチャ規約.md`](インフラストラクチャ規約.md) 参照）。

## ファイル参照保持（BookmarkData）

- ファイル / フォルダはコピーせず **Security Scoped Bookmark（BookmarkData）** で参照保持する
- アクセス時は `startAccessingSecurityScopedResource()` / `stopAccessingSecurityScopedResource()` を必ず対で呼ぶ
- 元ファイルが解決不能（削除）になった場合、Service が検知して Shelf から自動削除する
- BookmarkData の生成・解決は Repository（または Platform 層のファイルアクセサ）に閉じ込め、上位層は `URL` とドメインモデルのみを扱う

## マイグレーション

- スキーマは `user_version`（PRAGMA）等でバージョン管理し、起動時に順次マイグレーションを適用する
- マイグレーションは前方向き（forward-only）。各バージョンの変更を冪等に適用できるようにする
