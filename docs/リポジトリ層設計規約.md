# {{PROJECT_NAME / 例：〇〇プロジェクト}} Repository レイヤー設計方針

データベース（{{DB_SERVICE / 例：Supabase}}）への直接アクセスは `{{REPOSITORY_DIR / 例：server/repositories/}}` に集約します。
ローダー、アクション、サービス（UseCase）からは必ず Repository 関数を呼び出してデータを操作します。

## {{BASIC_PATTERN_TITLE / 例：基本パターン}}

```typescript
// {{PATTERN_PATH / 例：server/repositories/{feature}-repository.ts}}
import '{{SERVER_ONLY / 例：server-only}}';
import { {{DB_CLIENT_IMPORT / 例：createAdminClient}} } from '{{DB_LIB_PATH / 例：@mirai-gikai/supabase}}';

export async function {{DB_FUNCTION_NAME / 例：findFeatureById}}(id: string) {
  const {{DB_CLIENT_VAR / 例：supabase}} = {{DB_CLIENT_IMPORT}}();
  const { data, error } = await {{DB_CLIENT_VAR}}
    .from('{{TABLE_NAME / 例：features}}')
    .select('*')
    .eq('id', id)
    .single();

  if (error) {
    throw new Error(`{{ERROR_MESSAGE_PREFIX / 例：Failed to fetch feature:}} ${error.message}`);
  }
  return data;
}
```

## 実装ルール

- **`import '{{SERVER_ONLY / 例：server-only}}'`** を必ず先頭に付与してクライアントへの流出を防ぐ
- **データベースクライアントの生成（`{{DB_CLIENT_IMPORT / 例：createAdminClient()}}` 等）** は Repository 内に閉じ込め、ローダーやアクション、サービスから直接呼ばない
- Repository 関数は「DB クエリ実行 + 第一レイヤーのエラーハンドリング」のみを担当し、**複雑なビジネスロジックは呼び出し元（UseCaseなど）に残す**
  - 例外として、DBで直接実行されるべき集計処理などはリポジトリ側で実装しても良い
  - キャッシュ制御や細かな認証チェック、データのDTO変換はアプリケーション層以上で行う
