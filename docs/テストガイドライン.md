| ユニットテスト | `{{UNIT_TEST_DIR_PATTERN / 例：web/src/**/*.test.ts}}` | {{UNIT_TEST_TOOL / 例：Vitest, Jest}} | `{{UNIT_TEST_CMD / 例：pnpm test}}` | 純粋関数・ユーティリティのテスト | {{STATUS_UNIT_TEST / 例：導入済み}} |

## CI実行フロー

PR が作成されると、`{{CI_FILE / 例：code_check.yml}}` で以下が順次実行されます。

```text
PR Created / Push to main
  └── {{CI_JOB_NAME / 例：lint-and-typecheck}}
      ├── {{LINT_CMD / 例：pnpm run lint}}（フォーマット + lint）
      ├── {{TYPECHECK_CMD / 例：pnpm run typecheck}}（TypeScript 型チェック）
      └── {{TEST_CMD / 例：pnpm run test}}（全ワークスペースのユニットテスト実行）
```

全ステップがパスするとマージ可能になります。

## ユニットテスト

### 対象
- 外部依存のない純粋関数（`{{LIB_DIR / 例：lib/}}`、`{{UTILS_DIR / 例：utils/}}` など）
- {{DOMAIN_LOGIC_EXAMPLE_1 / 例：計算ロジック、文字列処理、データ変換等}}

### 配置ルール
実装ファイルと同じディレクトリに `*.test.ts` （または `*.spec.ts`）として配置します。

```text
{{FRONTEND_DIR / 例：web/src/}}
├── lib/
│   └── {{domain}}/
│       ├── {{target-file}}.ts
│       └── {{target-file}}.test.ts    ← 同階層に配置
└── features/
    └── {{feature-name}}/utils/
        ├── {{util-file}}.ts
        └── {{util-file}}.test.ts
```

### ポイント
- **mock は極力使わない**: 対象のロジックを純粋関数として切り出してからテストする。
- **ローカルサービスは real で動かす**: データベース等のローカルで起動できるサービスはモックせず、実際のローカルインスタンスに接続してテストする。
- **外部 API は DI でモックする**: {{EXTERNAL_API_EXAMPLE / 例：OpenAIなど}} の外部APIクライアントはインターフェースを定義し、テストでは Fake/Mock 実装に差し替える。

## DBロジックテスト（{{STATUS_DB_TEST}}）

### 対象
- {{DB_SERVICE / 例：Supabase}} のアクセス制御（例: RLSポリシー）
- DB関数（トリガーやストアドプロシージャなど）

### 前提
- ローカルDB環境が起動していること（`{{LOCAL_DB_START_CMD}}`）
- シードデータが投入済みであること（`{{SEED_CMD}}`）

### 設計方針
権限の異なるクライアント（管理者権限、匿名権限、認証済みユーザー権限など）を使い分けてテストします。セキュリティルールのテストでは「見えるべきデータが見える」「見えてはいけないデータが見えない」の両方を検証します。

## 統合テスト（{{STATUS_INTEG_TEST}}）

### 対象
- Server Actions や API Routes
- ビジネスロジック層（UseCase や Service 層）
- 実際のローカルDBに対して操作を行うシナリオテスト

### 設計原則
#### 1. 依存関係（DBクライアント等）を DI する
サービス関数は DBクライアントや 外部APIクライアント をパラメータとして受け取る設計にする。

#### 2. 外部 API は Fake で差し替える
外部 API との通信部分は、テスト用の Fake 実装を注入して利用する。

#### 3. DB 状態を必ず検証する（最重要）
**関数の戻り値だけでなく、DB の状態を必ず検証する。** これにより、実際にデータが永続化されたことを保証する。

```typescript
// 良い例: DB状態も検証
const result = await {{useCaseFunction}}(testClient, { ... });
expect(result.success).toBe(true);

const record = await fetchRecordFromDB(id);
expect(record.status).toBe("approved");  // DBに反映されたことを確認
```

#### 4. エラーケースの DB 状態検証
エラーが返された場合に、意図しないデータが作成・変更されていないことを DB レベルで確認する。

#### 5. クリーンアップを徹底する
テストで作成したデータは `afterEach` 等で必ず削除する。

## テスト作成時の注意事項

### テスト追加が必要なケース
以下のケースではテスト追加が必須です：
- {{MUST_TEST_CASE_1 / 例：金額計算ロジックの変更}}
- {{MUST_TEST_CASE_2 / 例：複雑なデータ変換ロジックの変更}}

加えて、以下のケースでもテスト追加を推奨します：
- 新しいユーティリティ関数の追加
- 既存ロジックのリファクタリング（回帰テストとして）
- バグ修正（再発防止テストとして）

### describe/it の構造化
関連するケースは `describe` でグループ化し、意図を明確にします。

### 一意性の確保
テストデータの ID やメールアドレスには `Date.now()` や ランダムなUUID などを含めて、並列実行時などの重複・コンフリクトを防止します。

### PR 前の確認
```bash
# 開発中に手元で実行・確認
{{UNIT_TEST_WATCH_CMD / 例：pnpm test:watch}}

# PR 作成前に全テストを実行
{{TEST_ALL_CMD / 例：pnpm test}}
```
