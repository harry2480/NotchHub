# NotchHub テストガイドライン

NotchHub（Swift / macOS）におけるテストの書き方・運用方法をまとめたガイドライン。基本ルールは `AGENTS.md` / `CLAUDE.md` にも記載する。

## テストの種類と構成

| 種類 | 配置場所 | フレームワーク | 実行コマンド | 用途 | 状態 |
|------|----------|----------|-------------|------|------|
| ユニットテスト | `Tests/NotchHubTests/` | XCTest / Swift Testing | `xcodebuild -scheme NotchHub test` | ドメインモデル・ViewModel・Service のロジック | 導入 |
| 永続化テスト | `Tests/NotchHubPersistenceTests/` | XCTest + 一時 SQLite | 同上（フィルタ指定） | Repository 実装（実 SQLite に対する CRUD） | 段階導入 |
| 統合テスト | `Tests/NotchHubIntegrationTests/` | XCTest | 同上 | Service + Repository + Platform Stub のシナリオ | 段階導入 |
| UI テスト | `Tests/NotchHubUITests/` | XCUITest | 同上 | クリティカルなノッチ操作フロー（任意） | 任意 |

## CI 実行フロー

PR 作成 / push 時に以下を順次実行する。

```text
PR Created / Push to main
  └── build-and-test
      ├── SwiftFormat --lint（フォーマット検証）
      ├── SwiftLint（静的解析）
      ├── xcodebuild build（ビルド）
      └── xcodebuild test（ユニット + 永続化 + 統合テスト）
```

全ステップがパスするとマージ可能になる。

## ユニットテスト

### 対象
- ドメインモデル（`Core/Models`）の生成・検証・変換ロジック
- ViewModel の状態遷移（Service を Stub 化）
- Service のビジネスロジック（Repository / Platform を Stub 化）
  - 例: Shelf 上限超過時の「最古の未ピン留めから削除」判定、寿命（7日/30日/カスタム）判定、最小状態の優先度決定

### 配置ルール
ソース構造を mirror して `Tests/NotchHubTests/` に配置する。

```text
Tests/NotchHubTests/
├── Core/Models/ShelfItemTests.swift
├── Features/Shelf/ShelfViewModelTests.swift
├── Features/Notch/NotchStatePriorityTests.swift
└── Services/ShelfServiceTests.swift
```

### ポイント
- **依存は protocol で注入し Stub に差し替える。** Service / Repository / Platform はすべて protocol 化されているので、ViewModel・Service テストでは Stub を渡す
- **純粋ロジックを切り出してからテストする。** OS API に依存する部分は Platform protocol の裏へ隠し、本体ロジックを純粋に保つ
- **`@MainActor` の ViewModel は `@MainActor` テストで検証する**

## 永続化テスト

### 対象
- `SQLiteShelfRepository` 等、SQLite に対する実 CRUD
- マイグレーション（`user_version` の前進適用）

### 前提
- 一時ディレクトリに使い捨ての SQLite ファイルを作り、テストごとに初期化する（本番の `~/Library/Application Support/NotchHub/` は使わない）

### 設計方針
- 戻り値だけでなく、**DB の状態（行の有無・並び順・ピン留めフラグ）を必ず検証する**
- BookmarkData の解決は実ファイル（一時ファイル）を作って検証し、削除時の自動除去も確認する

---

## 統合テスト

### 対象
- Service + Repository（実 SQLite）+ Platform Stub を組み合わせたシナリオ
  - 例: 「ファイルを Shelf へ追加 → 検索でヒット → 元ファイル削除 → 自動的に Shelf から消える」

### 設計原則
1. **依存を DI する** — Service は Repository / Platform を init で受け取る
2. **OS API（AirDrop・Share・EventKit・AppleScript）は Stub に差し替える** — 実際の共有 UI を出さずに副作用呼び出しを検証
3. **状態を必ず検証する** — 戻り値 + DB / インメモリ状態の両方を確認
4. **クリーンアップを徹底する** — `tearDown` で一時ファイル・一時 DB を削除する

---

## UI テスト（任意）

### 対象
- クリティカルなノッチ操作フロー（ドラッグ → 3 Zone → Drop → Toast）

### 設計方針
- XCUITest を使用。CI では本数を絞り、不安定化を避ける
- ノッチウィンドウ（`NSPanel`）の特性上、自動化が難しい部分は手動チェックリストで補う

---

## テスト作成時の注意事項

### テスト追加が必須のケース
- Shelf の上限超過・寿命判定ロジックの変更
- 最小状態の優先度ロジックの変更
- Repository のクエリ・マイグレーションの変更

### 推奨ケース
- 新しい Service / ViewModel ロジックの追加
- リファクタリング（回帰テストとして）
- バグ修正（再発防止テストとして）

### 構造化
- 関連ケースは `XCTestCase` のサブクラス、または Swift Testing の `@Suite` でグループ化し意図を明確にする

### 一意性の確保
- 一時ファイル・一時 DB のパスにはテスト固有のサフィックス（`UUID`）を含め、並列実行の衝突を防ぐ

### PR 前の確認
```sh
# ビルド + 全テスト
xcodebuild -scheme NotchHub test

# 特定ターゲットのみ
xcodebuild -scheme NotchHub test -only-testing:NotchHubTests
```
