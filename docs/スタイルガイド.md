# NotchHub スタイルガイド

Swift / SwiftUI コードのスタイルおよびノッチ UI の表示ルール。

## 1. Swift コーディングスタイル

### ルール

- フォーマットは SwiftFormat / Swift 標準スタイルに従い、インデントは 4 スペース
- 命名: 型は `UpperCamelCase`、プロパティ・関数・変数は `lowerCamelCase`、定数も `lowerCamelCase`
- ファイル名は主たる型名と一致させる（`ShelfViewModel.swift` 等）
- `force unwrap`（`!`）・`try!`・`as!` は原則禁止。`guard let` / `if let` / `Result` / `throws` で扱う
- 値型（`struct` / `enum`）を優先し、参照型（`class`）は同一性や寿命管理が必要な場合のみ使う
- アクセス修飾子は最小公開原則。外部に出さないものは `private` / `fileprivate` に閉じる

### 理由

ノッチ UI は常駐し OS の機微 API を触るため、クラッシュ（force unwrap 起因）を構造的に排除し、状態を予測可能に保つ必要があるため。

## 2. SwiftUI ビュー

### ルール

- View は小さく分割し、`body` は宣言的に保つ。複雑な分岐・整形は ViewModel か算出プロパティへ逃がす
- 状態は ViewModel（`@Observable` / `ObservableObject`）に集約し、View に `@State` を増やしすぎない（純粋にローカルな UI 状態のみ `@State`）
- マジックナンバー（サイズ・余白）はハードコードせず、定数（`enum Layout` 等）に集約する
- 色・フォントは直接指定せず、アセットカタログのカラーセット / セマンティックカラー（`Color.primary` 等）経由で参照し、ダークモードに追従させる

### 具体例

```swift
// NG: マジックナンバー・直値カラー
.frame(width: 560, height: 360)
.foregroundColor(Color(red: 0.1, green: 0.45, blue: 0.9))

// OK: 定数・セマンティックカラー
.frame(width: NotchLayout.expanded.width, height: NotchLayout.expanded.height)
.foregroundColor(.accentColor)
```

## 3. ノッチ表示ルール

### サイズ定数（擬似ノッチ）

ノッチ／擬似ノッチのサイズは状態ごとに定数化する。

| 状態 | サイズ |
|---|---|
| 通常 | 180 x 32 |
| ドラッグ | 520 x 120 |
| クリック | 560 x 360 |

```swift
enum NotchLayout {
    static let collapsed = CGSize(width: 180, height: 32)
    static let dragging  = CGSize(width: 520, height: 120)
    static let expanded  = CGSize(width: 560, height: 360)
}
```

### 最小状態の表示

- 通常時は何も表示せずノッチに同化させる
- 最小状態に表示する情報は優先度順で **常に 1 つだけ**（AI 承認待ち > ドラッグ中 > AirDrop/Share 実行中 > メディア再生中 > 予定 10 分前）
- 展開トリガーは「ドラッグ接近 / クリック / AI 承認待ち」のみ。Hover・カーソル接近では展開しない（誤操作防止）

## 4. 非同期・並行処理

### ルール

- 非同期処理は `async / await` を基本とし、コールバック地獄を避ける
- UI 更新は必ずメインアクターで行う（`@MainActor` を ViewModel に付与）
- 共有可能な可変状態は `actor` で保護するか、メインアクターに閉じる
- ファイル監視（File System Events）・Socket・グローバルイベント監視はバックグラウンドで動かし、結果をメインアクターへ橋渡しする

---

## 5. ログ・エラー

- ログは `os.Logger`（統合ログ）を使い、`print` をプロダクションコードに残さない
- ユーザー向けエラーは Toast で簡潔に伝え、詳細はログへ。サイレント失敗（握りつぶし）を避ける
