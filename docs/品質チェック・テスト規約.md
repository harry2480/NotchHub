# NotchHub 品質チェック規約

SwiftFormat + SwiftLint + Swift コンパイラ（型チェック）+ XCTest によるコード品質チェック規約。

---

## Quality Check 一覧

| チェック | ツール | 対象 | コマンド |
|---|---|---|---|
| フォーマット | SwiftFormat | 全ソース | `swiftformat --lint .` |
| 静的解析 / Lint | SwiftLint | 全ソース | `swiftlint` |
| ビルド（型チェック含む） | Swift コンパイラ | 全ソース | `xcodebuild -scheme NotchHub build` |
| ユニットテスト | XCTest / Swift Testing | Model / ViewModel / Service | `xcodebuild -scheme NotchHub test` |
| 永続化・統合テスト | XCTest | Repository / シナリオ | `xcodebuild -scheme NotchHub test` |

### verify（品質チェック一括）実行フロー

push 前に以下の順で実行し、全パスを確認する：

1. `swiftformat --lint .`（フォーマット検証）
2. `swiftlint`（静的解析）
3. `xcodebuild -scheme NotchHub build`（ビルド・型チェック）
4. `xcodebuild -scheme NotchHub test`（テスト）

> これらを 1 コマンドにまとめるため、`Makefile` の `make verify` または `scripts/verify.sh` を用意することを推奨する。

---

## 型・コンパイル

- **警告ゼロを維持する。** コンパイラ警告が残る PR はマージ不可とする
- **`!`（force unwrap）・`try!`・`as!` は原則禁止。** やむを得ない場合は理由をコメントで明記する
- Swift Concurrency の警告（データ競合・アクター越境）を抑制せず、設計で解消する

---

## 静的解析（SwiftLint）

- 共通ルールは `.swiftlint.yml` で管理する
- レイヤー違反（View からの Service/Repository 直接参照など）はレビューで検出する。可能なら `custom_rules` で機械的に弾く
- ルール無効化（`// swiftlint:disable`）は範囲を最小化し、理由を併記する

---

## 依存方向の検証

- [`アーキテクチャ.md`](アーキテクチャ.md) の依存方向（`View → ViewModel → Service → Repository/Platform protocol`、Model は最内層）を守る
- Feature を Swift Package のローカルターゲットへ分割し、`import` 可能範囲をコンパイル時に制約することで依存違反を機械的に防ぐ
- 違反があった場合は型の配置・protocol の切り出しを見直す

---

## テスト方針

- ロジックは Stub を用いた Unit テストで担保する（[`テストガイドライン.md`](テストガイドライン.md) 参照）
- Repository は一時 SQLite に対する永続化テストで担保する
- OS API（AirDrop / Share / EventKit / AppleScript / Accessibility）は Platform protocol の裏に隠し、Stub で副作用呼び出しを検証する
