## Quality Check 一覧

| チェック | ツール | 対象 | コマンド |
|---|---|---|---|
| Unit テスト | {{TEST_TOOL}} | {{UNIT_TEST_TARGET / 例：domain, application}} | `{{CMD_TEST_UNIT / 例：pnpm test:unit}}` |

### {{VERIFY_COMMAND_NAME / 例：verify コマンド}} 実行フロー

`{{CMD_VERIFY / 例：pnpm verify}}` は以下の順序で実行されることを期待する：

1. {{STEP_LINT / 例：lint}}
2. {{STEP_GENERATE / 例：prisma generate（型生成）}}
3. {{STEP_TYPECHECK / 例：typecheck}}
4. {{STEP_TEST / 例：unit test}}
5. {{STEP_DEPCRUISE / 例：depcruise}}

コード変更後は必ず実行し、全パスすることを確認する。

## 型チェック (TypeCheck)

- `{{TYPE_CMD / 例：tsc --noEmit}}` で型エラーがないことを確認する
- **`any`** の使用は原則禁止（一時的な回避にはlint抑制コメントを明確に併記する）
- CI で必ず実行。型エラーがある PR はマージ不可とする

---

## 依存方向チェック (Dependency Cruise)

- `{{ARCHITECTURE_DOC / 例：アーキテクチャ.md}}` で定義された依存方向（例：presentation → application → domain ← infrastructure）をチェックする
- 違反があった場合、モジュールの切り出し・配置を見直す
