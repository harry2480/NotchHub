# NotchHub 開発環境メモ

[`インフラストラクチャ規約.md`](インフラストラクチャ規約.md) では Xcode / `xcodebuild` を前提とした記述があるが、本リポジトリは規約が許容する **Swift Package Manager（`Package.swift`）構成** を採用している。理由と、ローカル検証手順を記す。

## なぜ SPM 構成か

- `xcodebuild` は full Xcode を要求するが、Command Line Tools のみの環境でも開発・CI を回せるようにするため、`Package.swift` ベースとした（[`アーキテクチャ.md`](アーキテクチャ.md)・[`インフラストラクチャ規約.md`](インフラストラクチャ規約.md) はいずれも SPM 構成を明示的に許容している）。
- 配布用の `.app` バンドル化・署名・公証の段になったら、この SPM 製バイナリを bundle にラップする（`Resources/Info.plist`・`Resources/NotchHub.entitlements` が雛形）。

## パッケージ構成

```text
Package.swift            # macOS 14+ / executable: NotchHub, testTarget: NotchHubTests
Sources/NotchHub/        # アプリ本体（App / Core / Features / Services / Repositories / Platform）
Tests/NotchHubTests/     # swift-testing（ソース構造を mirror）
Resources/               # Info.plist・entitlements（バンドル化時に適用）
scripts/                 # verify / swift-test / swiftlint ラッパー
```

- 実行は menu-bar / notch エージェント（`LSUIElement` / `.accessory`）なので、SwiftUI `App` ライフサイクルではなく AppKit の `NSApplication` を直接駆動する（`NSPanel` 制御のため）。

## テストフレームワーク

XCTest は full Xcode にのみ同梱されるため、本リポジトリのテストは **swift-testing（`import Testing`）** を使う。CLT には `Testing.framework` が含まれるが、`swift test` が探索パスを通さないため、ラッパースクリプトで framework search path / rpath を補う。

## 品質チェック（verify）

[`品質チェック・テスト規約.md`](品質チェック・テスト規約.md) の verify を SPM 向けにマップしたもの。push 前に必ず実行する。

```sh
make verify          # 下記を順に実行
# 1. swiftformat --lint .
# 2. ./scripts/swiftlint.sh --strict     # CLT では DYLD_FRAMEWORK_PATH を補完
# 3. swift build                         # 警告ゼロ
# 4. ./scripts/swift-test.sh             # CLT では Testing.framework のパスを補完
```

個別コマンド:

```sh
make build           # swift build
make test            # ./scripts/swift-test.sh
make format          # swiftformat .
make lint            # ./scripts/swiftlint.sh --strict
```

## CLT 専用ワークアラウンドの所在

full Xcode 環境ではラッパーは何もせず素の `swift test` / `swiftlint` を呼ぶ。CLT 環境でのみ次を補う:

- `scripts/swift-test.sh` — `Testing.framework` と `lib_TestingInterop.dylib` の `-F` / `-rpath`
- `scripts/swiftlint.sh` — `sourcekitdInProc.framework` を読むための `DYLD_FRAMEWORK_PATH`
