# NotchHub UI（SwiftUI）アーキテクチャ実装方針

## 1. 概要

NotchHub の UI 層（SwiftUI ビュー + ViewModel）の実装方針を定める。Feature ベースのモジュール構成を採用し、ノッチを中心とした常駐 UI を保守性高く構築する。

> ここでの「フロントエンド」は、ネイティブ macOS アプリの **UI 層（SwiftUI ビュー + ViewModel）** を指す。バックエンドに相当するのは Service / Repository / Platform 層（[`アーキテクチャ.md`](アーキテクチャ.md) 参照）。

## 2. 技術スタック

### コア技術
- **言語**: Swift
- **UI フレームワーク**: SwiftUI（コンテンツ）+ AppKit（ノッチウィンドウ・グローバルイベント）
- **状態管理**: `@Observable` / `ObservableObject` による ViewModel
- **ウィンドウ**: `NSPanel`（非アクティブ化・最前面）でノッチ／擬似ノッチを描画
- **永続化アクセス**: Service 経由（View から直接 DB/OS API を触らない）

### 開発ツール
- **フォーマッタ / Linter**: SwiftFormat / SwiftLint
- **テスト**: XCTest / Swift Testing + ViewInspector 等（必要に応じて）

## 3. ディレクトリ構成

```text
Sources/NotchHub/Features/
├── Notch/
│   ├── Views/              # NotchRootView, CollapsedNotchView, ExpandedNotchView
│   ├── ViewModels/         # NotchViewModel（展開状態・最小状態の優先度制御）
│   └── DropZone/           # 3 Zone（Shelf/Share/AirDrop）の Drop 受け
├── Shelf/
│   ├── Views/              # ShelfListView, ShelfItemRow, ShelfSearchBar
│   └── ViewModels/         # ShelfViewModel
├── AirDrop/
│   ├── Views/              # AirDropHistoryView
│   └── ViewModels/         # AirDropViewModel
├── Share/
│   └── ViewModels/         # ShareViewModel
├── AIMonitor/
│   ├── Views/              # AISessionListView, ApprovalPromptView
│   └── ViewModels/         # AIMonitorViewModel
├── Calendar/
│   ├── Views/              # CalendarView（Next / Today）
│   └── ViewModels/         # CalendarViewModel
└── Media/
    ├── Views/              # MediaControlView
    └── ViewModels/         # MediaViewModel
```

共通 UI（Toast・アイコン・タブバー）は `Core/` 配下の共有コンポーネントへ置く。

## 4. 機能要件（UI 観点）

### Phase 1: MVP

#### 4.1 ノッチ + ドラッグ UI
- 通常時はノッチに同化、明示トリガーで展開
- ドラッグ接近で 3 Zone（左 Shelf / 中央 Share / 右 AirDrop）を表示し、Drop 時のみ実行
- Dead Zone・Drop 前強調・Toast・Undo で誤操作を防止

#### 4.2 Shelf タブ
- ピン留め → 新しい順で一覧表示（アイコン + 名前）
- 全文検索バー（ファイル名・フォルダ名・URL・本文）
- 行操作: 開く / Finder 表示 / 再ドラッグ / AirDrop / Share / ピン留め / 削除

#### 4.3 タブ構成
- `Shelf | Calendar | Media | AI`（初期タブ Shelf、設定で変更可）
- 右上 ⚙ メニュー: Settings / Launch at Login / Quit App

### Phase 2: 補助機能の拡充
- AI Monitor の承認待ち自動展開・ログ表示
- Calendar / Media タブの表示最適化

## 5. 非機能要件（UI）

### パフォーマンス
- ノッチ展開アニメーションは滑らかに（60fps 目標）。展開／収縮のレイテンシを体感ゼロに近づける
- 常駐時の CPU / メモリ使用を最小化（アイドル時はイベント駆動でほぼ無負荷）
- サムネイル・アートワークはキャッシュ（`cache/`・`thumbnails/`）して再計算を避ける

### アクセシビリティ
- VoiceOver 対応（アイテム・操作に適切なラベル）
- キーボード操作対応（Esc で閉じる等。主導線はマウス／ドラッグ）

## 6. 実装ガイドライン

### View 設計
1. 単一責任。View は表示と入力に専念し、状態は ViewModel に集約
2. プレビュー（`#Preview`）で各状態（通常・ドラッグ・展開・承認待ち）を確認可能にする
3. ノッチ／擬似ノッチの差異はレイアウト定数で吸収し、View ロジックを分岐させすぎない

### アーキテクチャ原則
1. **Feature ベース構造**: 機能ごとに Views / ViewModels をモジュール化
2. **責任分離**: View ⇄ ViewModel ⇄ Service の一方向依存を守る
3. **テスタビリティ**: ViewModel は Service を protocol で受け取り、Stub でテスト可能にする

## 7. 実装順序

要件定義の実装優先順位（[`要件定義.md`](要件定義.md) 第24章）に従う。

1. Notch Core（ノッチウィンドウ・展開制御）
2. Drag UI（3 Zone・Drop 検出）
3. Shelf（一覧・保存）
4. Search
5. AirDrop / Share / Screenshot
6. AI Monitor / Calendar / Media

## 8. テスト戦略

- ViewModel のロジックは Unit テスト（Service を Stub 化）
- View は `#Preview` と必要に応じたスナップショット/インスペクションで確認
- 詳細は [`テストガイドライン.md`](テストガイドライン.md) を参照
