# docs/ テンプレート化計画

以下は docs ディレクトリのテンプレ化（汎用化）方針の整理です。目的は
「プロジェクト非依存のドキュメントをテンプレート化」して再利用性を高めることです。

## 対応方針（3分類）

- そのまま残す（作業不要）: 汎用的でプロジェクト非依存のファイル
- 小〜中修正でテンプレ化: プレースホルダー化や一部文章修正でテンプレ化可能なファイル
- 削除（またはアーカイブ）: 完全にプロジェクト固有でテンプレ化が不適切なファイル

---

## 1) そのまま残す（即テンプレ利用可）

| ファイル | 内容（要約） |
|---|---|
| architecture.md | DDD 4層設計規約 |
| frontend.md | Next.js + shadcn/ui + Tailwind 規約 |
| infrastructure.md | pnpm monorepo / Vercel / Supabase / Prisma 構成 |
| quality.md | Vitest / tsc / dependency-cruiser による品質規約 |
| 20260220_1530_スタイルガイド.md | Tailwind dvh ルール（汎用） |

> 備考: 上記はプロジェクト名や固有パスを含まないため、そのままテンプレとして残します。

---

## 2) 小〜中修正でテンプレ化（想定 4件）

下表は修正方針のサマリです。修正は主にプレースホルダー化（<project> 等）とコード例の汎用化を行います。

| ファイル | 修正ポイント |
|---|---|
| 20260301_0700_開発用UIプレビュー環境.md | `features/bills/` や `BillCard` のような固有コンポーネント名を `features/<feature>/` / `<YourComponent>` に置換 |
| 20260219_1000_テストガイドライン.md | 冒頭のプロジェクト固有表記を削除し、関数名やパス（例: `calculateAiCost`）を汎用プレースホルダーに置換 |
| repository-layer.md | `@mirai-gikai/supabase` を `@<project>/db` に、`bills` テーブル名を `<entity>` に抽象化 |
| 20260407_1800_adminGoogleログイン設定手順.md | アプリ名・ポート・ロール名を `<project-name>` / `<PORT>` に置換 |

> 実作業: 各ファイルを開き、正規表現で固有文字列を置換した上で目視レビューを行います。

---

## 3) 削除（アーカイブ推奨）

- 対象: 完全にプロジェクト固有で、他プロジェクトでの再利用価値が低いファイル群（現在分: 23件想定）
- 方針: リポジトリ内に `docs/archive/` を作成して移動するか、別ブランチへ退避します。

> 例: 20250904〜20251221 のみらい議会固有仕様や、特定時点のスナップショット資料など。

## 実施スコープと作業フロー（短縮版）

1. 現状ファイルリストの確定（完了）
2. 小〜中修正ファイルの実作業（置換→目視）
3. 修正ファイルを `docs/` に反映し、差分を PR で提出
4. 削除対象は `docs/archive/` に移動して PR を作成

## 想定スケジュール（目安）

| ステップ | 工数目安 |
|---:|---:|
| 現状調査（完了） | 0.5日 |
| 修正設計（プレースホルダー設計） | 0.5日 |

| 削除/アーカイブ作業 | 0.5日 |
